# Donggri Ledger Build Story Draft

Notion 하위페이지로 옮기기 위한 Step 1~5 장문 원고입니다.  
각 Step은 `문제 -> 왜 그렇게 구현했는지 -> 실제 구현 -> 예시 -> 기능 확인 -> 다음 단계` 흐름으로 정리했습니다.  
한 페이지당 실제 블로그 초안처럼 읽히도록 내용을 확장했고, Notion에 그대로 붙여도 되는 형태로 맞췄습니다.

## Step 1. 로컬 FastAPI + SQLite로 개인 가계부 MVP 만들기

### 추천 제목

- 로컬 FastAPI + SQLite로 개인 가계부 MVP를 먼저 만든 이유

### 검색 유입 키워드

- local ledger app
- fastapi budget tracker
- sqlite finance dashboard
- 로컬 가계부 만들기
- 개인 재무 대시보드

### 글 구조

#### 문제

가계부를 만들겠다고 마음먹었을 때 가장 먼저 부딪히는 문제는 기능 부족이 아니라 범위 과다입니다. 처음부터 서비스형으로 가면 로그인, 배포, 외부 접속, 사용자 분리, 도메인, 운영 문서까지 한 번에 터집니다. 그렇게 되면 정작 중요한 “내가 이 입력 흐름을 매일 쓸 것인가”를 확인하기도 전에 프로젝트가 지칩니다. 그래서 첫 단계에서는 운영성보다 사용감을 검증하는 쪽으로 방향을 잡았습니다. 빠르게 실행하고, 입력하고, 저장하고, 다시 열었을 때 흐름이 자연스러운지를 보는 게 우선이었습니다.

#### 왜 이렇게 구현했는가

FastAPI는 API와 화면을 함께 붙여보기에 속도가 빠른 편이고, SQLite는 별도 인프라 없이 즉시 시작할 수 있다는 장점이 있습니다. MVP 단계에서는 데이터가 많지 않고, 사용자도 한 명뿐이기 때문에 복잡한 인프라보다 “바로 돌아가는가”가 더 중요합니다. 특히 월별 요약 카드나 거래 입력 폼은 코드를 예쁘게 짜는 것보다 손에 익는 구성이 먼저였습니다. 그래서 이 단계에서는 로컬 DB 저장, 거래 등록, 자산 모델, 카테고리 구조 정도만 먼저 잡고 실제 생활비 입력 감각을 확인하는 데 집중했습니다.

#### 구현 포인트

- 자산, 거래, 카테고리, 예산의 가장 기본적인 데이터 구조 설계
- 월별 수입/지출/잔액 요약 카드 배치
- 거래 입력 후 즉시 목록 반영
- 로컬 SQLite 저장으로 데이터 유지
- FastAPI로 API와 정적 화면을 한 흐름으로 연결

#### 예시

- 은행 자산 1개를 만들고 월급 1건, 식비 1건을 입력했을 때 월 합계가 바로 반영되는지 확인
- 자산 잔액이 눈으로 봤을 때 실제 통장 흐름과 크게 다르지 않은지 체크
- 프로그램을 껐다 켜도 데이터가 그대로 남는지 확인

#### 기능 확인

- 수입 / 지출 입력이 즉시 반영되는가
- 월별 합계가 계산과 실제 체감에 맞는가
- 자산 잔액 흐름이 자연스러운가
- 거래 입력 폼이 너무 복잡하지 않은가

#### 다음 단계로 넘어간 이유

이 단계까지 오면 “입력 자체는 되는데 현실이 덜 반영된다”는 느낌이 생깁니다. 실제 생활에서는 카드 쓴 날과 돈 빠지는 날이 다르고, 고정지출은 매달 반복되고, 투자 자산은 일반 소비와 따로 떼어볼 수 없습니다. 그래서 다음 단계에서는 단순한 가계부가 아니라 실사용 가능한 생활비 관리 도구로 확장하는 작업이 필요했습니다.

## Step 2. 카드 결제예정, 고정지출, 투자까지 붙여 실사용 가계부로 넓히기

### 추천 제목

- 카드 결제예정과 투자 포트폴리오를 붙여 가계부를 실사용 단계로 확장한 방법

### 검색 유입 키워드

- credit card settlement planner
- fixed expense tracker
- portfolio tracker in ledger
- 카드 결제예정 가계부
- 투자 포트폴리오 가계부

### 글 구조

#### 문제

일반 가계부를 며칠만 써보면 바로 드러나는 문제가 있습니다. 카드 결제는 쓴 날과 실제 돈이 빠져나가는 날이 다르기 때문에 “이번 달 잔액”이 자주 왜곡됩니다. 여기에 고정지출과 예산이 빠지면 다음 달 부담을 미리 보기 어렵고, 투자 포트폴리오를 다른 앱에서 보면 전체 자산 흐름이 끊깁니다. 결국 실사용 단계로 가려면 소비, 정산 예정, 반복지출, 투자까지 한 화면에서 이어져야 했습니다.

#### 왜 이렇게 구현했는가

핵심은 카드 결제일과 결제통장을 거래가 아니라 자산 속성으로 다루는 것이었습니다. 이렇게 하면 사용자는 결제 수단만 고르면 되고, 실제 정산 계산은 시스템이 일관되게 할 수 있습니다. 고정지출은 월별 생활비 압박을 미리 보여주는 역할을 하고, 예산은 현재 소비 속도를 제어해줍니다. 투자 계좌를 별도 앱이 아니라 같은 가계부 안에 넣은 이유도 명확합니다. 소비와 투자 흐름을 분리해 보면 편해 보이지만, 결국 내 돈은 한 덩어리이기 때문에 자산 전체 그림을 한 번에 보는 쪽이 더 실용적이었습니다.

#### 구현 포인트

- 카드 자산별 결제일 / 결제통장 연결
- 카드 사용 시 실제 출금일과 결제예정 분리
- 고정지출 등록과 활성화 관리
- 카테고리별 예산 설정
- 투자 계좌, 종목, 수익률 요약
- 자산 삭제 시 연결 데이터 검사

#### 예시

- 신용카드로 3월 12일 식비를 결제했는데 실제 출금은 4월 5일인 경우, 현재 소비에는 반영되지만 현금 잔액은 즉시 빠지지 않게 처리
- 통신비, 월세, 구독료 같은 고정지출을 미리 등록해 월 시작 시 부담을 가늠
- 주식과 코인을 따로 보지 않고 전체 자산 중 투자 비중이 얼마나 되는지 확인

#### 기능 확인

- 카드 사용 직후 현금 잔액이 즉시 빠지지 않는가
- 이번달 카드결제예정이 요약 카드에 보이는가
- 고정지출과 예산이 동시에 관리되는가
- 투자 원금, 평가금, 수익률이 계산되는가

#### 다음 단계로 넘어간 이유

기능이 늘어나면 프로젝트 설명도 함께 필요해집니다. 혼자 쓰는 도구일 때는 괜찮지만, 구조가 커지면 저장소를 처음 보는 사람도 “이게 어떤 앱인지” 바로 이해할 수 있어야 합니다. 그래서 다음 단계에서는 코드보다 설명력, 검색 유입, 첫 인상에 가까운 영역을 손보기 시작했습니다.

## Step 3. GitHub Pages 랜딩과 SEO 구조로 프로젝트 설명력을 높이기

### 추천 제목

- GitHub Pages로 프로젝트 랜딩 페이지와 검색 유입 구조를 만든 과정

### 검색 유입 키워드

- github pages seo
- software landing page example
- project showcase page
- 프로젝트 소개 페이지 만들기
- SEO 랜딩 페이지

### 글 구조

#### 문제

코드만 있는 저장소는 만든 사람에겐 익숙하지만, 처음 보는 사람에겐 불친절합니다. 특히 사이드 프로젝트는 “무엇을 만들었는지”, “왜 만들었는지”, “어떤 화면이 있는지”가 빠르게 전달되지 않으면 검색 유입도 약하고 공유도 어렵습니다. README만으로 모든 메시지를 전달하기엔 한계가 있었고, 실제 앱과 별도로 제품 소개 성격의 페이지가 필요했습니다.

#### 왜 이렇게 구현했는가

GitHub Pages는 이미 저장소와 연결돼 있어서 별도 인프라 없이 소개 페이지를 운영하기 좋습니다. 여기에 Open Graph 메타 태그, 구조화 데이터, sitemap, robots 같은 기본 SEO 구성을 붙이면 단순한 문서가 아니라 검색 유입용 랜딩 역할도 할 수 있습니다. 중요한 건 글을 길게 쓰는 것이 아니라, 첫 화면에서 핵심 가치와 스크린샷을 빠르게 보여주는 구성이었습니다. 그래서 서비스 소개, 기능 요약, 화면 예시, 구조 설명, FAQ를 중심으로 다시 짰습니다.

#### 구현 포인트

- GitHub Pages용 제품 소개 랜딩 페이지 구성
- Open Graph 메타 태그 설정
- 구조화 데이터와 sitemap 구성
- 실제 스크린샷 중심 섹션 배치
- FAQ와 간단한 서비스 구조 설명 추가

#### 예시

- 저장소에 처음 들어온 사람이 README를 열기 전에 GitHub Pages 첫 화면만 보고 “이건 Supabase와 Vercel로 운영하는 가계부 서비스구나”를 이해하도록 구성
- 스크린샷을 작은 카드처럼 흩뿌리지 않고, 실제 화면의 무드를 보여주는 큰 패널 중심으로 배치

#### 기능 확인

- GitHub Pages에서 주요 스크린샷이 잘 보이는가
- 검색엔진용 description, keywords, FAQ가 들어갔는가
- 저장소 README와 소개 페이지 메시지가 어긋나지 않는가
- 제품 설명과 코드 구조 설명이 따로 놀지 않는가

#### 다음 단계로 넘어간 이유

설명력은 좋아졌지만, 여전히 실서비스처럼 쓰기엔 로컬 서버 방식의 한계가 뚜렷했습니다. 외부 접속, 유지보수, 여러 기기 접근, 사용자 분리를 생각하면 구조를 바꿀 시점이 온 것이었습니다. 그래서 다음 단계에서 가장 큰 전환이 일어났습니다.

## Step 4. Supabase + Vercel로 서비스형 구조로 전환하기

### 추천 제목

- 로컬 FastAPI + SQLite 가계부를 Supabase + Vercel 서비스형 구조로 옮긴 이유

### 검색 유입 키워드

- supabase budget app
- vercel finance app
- migrate sqlite to supabase
- 가계부 supabase
- vercel 개인 서비스 배포

### 글 구조

#### 문제

로컬 서버를 계속 켜두는 방식은 한두 번 테스트할 때는 괜찮지만, 외부에서 접속하고 유지보수하려고 하면 갑자기 난이도가 확 올라갑니다. 웹훅, 포트, 배포, 집 PC 전원, 네트워크 문제까지 한꺼번에 따라옵니다. 여기에 사용자별 데이터 분리까지 붙이면 SQLite 기반 단일 로컬 구조로는 한계가 명확했습니다. 결국 “잘 돌아가는 로컬 앱”에서 “운영 가능한 서비스형 앱”으로 방향을 바꿔야 했습니다.

#### 왜 이렇게 구현했는가

Vercel은 정적 프론트 배포가 빠르고, GitHub push 기준으로 재배포가 단순합니다. Supabase는 Auth, Postgres, RLS, Edge Functions를 한 번에 관리할 수 있어서 작은 서비스에 잘 맞습니다. 특히 사용자별 데이터 분리가 필요한 순간부터는 DB 정책과 인증이 핵심인데, Supabase는 그 부분을 비교적 짧은 시간 안에 안정적으로 붙일 수 있었습니다. 그래서 프론트는 Vercel, 인증과 데이터는 Supabase, 코드와 문서는 GitHub로 역할을 분리하는 방향이 가장 현실적이었습니다.

#### 구현 포인트

- 정적 웹 앱 빌드 구조로 전환
- Supabase schema와 RLS 작성
- 사용자별 `user_id` 데이터 분리
- Edge Function 기반 회원가입 / 복구 / 시세 갱신 흐름 구성
- Vercel 환경변수와 빌드 설정 정리
- GitHub를 기준으로 배포 이력 관리

#### 예시

- 집 PC를 켜두지 않아도 외부에서 접속 가능
- 같은 계정으로 다른 기기에서 로그인 가능
- 다른 사용자의 데이터가 섞이지 않도록 `RLS` 정책으로 차단
- 정적 배포 결과물은 `dist/`, 실제 운영은 Vercel 주소로 접근

#### 기능 확인

- 회원가입과 로그인 흐름이 외부 배포 환경에서 도는가
- 사용자별 데이터가 섞이지 않는가
- 카드 / 예산 / 투자 데이터가 기존 흐름대로 유지되는가
- GitHub 변경이 Vercel 재배포와 연결되는가

#### 다음 단계로 넘어간 이유

서비스형 구조로 옮기고 나면 이제부터는 “돌아간다”보다 “사용자가 막히지 않는다”가 더 중요해집니다. 회원가입, 비밀번호 찾기, 함수 에러, 실시간 가격 새로고침처럼 운영 중 자주 마주치는 UX와 디버깅 포인트를 더 안정적으로 다듬을 필요가 생겼습니다.

## Step 5. 아이디 인증, 보안질문 복구, 실시간 시세 안정화까지 마무리하기

### 추천 제목

- 보안질문 기반 회원가입과 실시간 시세 새로고침을 안정화한 마지막 정리

### 검색 유입 키워드

- security question auth
- supabase edge function auth
- yahoo finance edge function
- 보안질문 비밀번호 찾기
- supabase edge function 오류 해결

### 글 구조

#### 문제

서비스형 구조로 바꾼 뒤에는 의외로 SMTP, 도메인, JWT, API key, 환경변수 같은 운영 문제가 더 크게 다가왔습니다. 이메일 기반 복구는 정석이지만, 초반 운영 입장에서는 도메인과 메일 세팅이 생각보다 번거로웠습니다. 게다가 Edge Function은 에러가 뭉뚱그려 보이면 어디서 막혔는지 추적하기 어렵습니다. 실시간 투자 시세도 잘 되면 좋지만, 실패할 때 원인을 모르면 신뢰도가 금방 떨어집니다.

#### 왜 이렇게 구현했는가

그래서 비밀번호 복구는 이메일 대신 보안질문 기반 흐름으로 단순화했습니다. 이 방식은 완벽한 정답은 아니지만, 초반 소규모 서비스 운영에서 SMTP 부담을 줄이기에 현실적이었습니다. 또 함수 에러는 숨기지 않고 실제 메시지를 최대한 드러내도록 바꿔서 디버깅 속도를 높였습니다. 실시간 가격 새로고침은 투자 탭의 신뢰도를 결정하는 기능이기 때문에, 호출 헤더와 함수 응답을 정리해 실패 원인을 더 빨리 파악할 수 있게 다듬었습니다.

#### 구현 포인트

- 아이디 + 비밀번호 + 보안질문 3개 기반 회원가입
- 비밀번호 찾기용 질문 불러오기와 재설정
- JWT verification 설정 정리
- API key / 프로젝트 매칭 점검
- 투자 시세 새로고침 호출 보강
- 함수 에러 본문 노출 개선
- 작은 마감 UX까지 수정

#### 예시

- 회원가입 시 질문 3개를 고르고 답변을 저장한 뒤, 나중에 이메일 없이도 비밀번호 재설정 가능
- `Invalid API key`, `column does not exist`, `non-2xx status` 같은 오류를 하나씩 분리해서 실제 원인 기준으로 수정
- 투자 관리 탭에서 삭제 버튼 아이콘처럼 작지만 신뢰도에 영향을 주는 마감 요소도 정리

#### 기능 확인

- 회원가입 성공
- 로그인 성공
- 비밀번호 재설정 성공
- 투자 가격 새로고침 성공
- 에러 메시지가 너무 추상적이지 않은가
- 삭제 버튼, 상단 액션, 인증 진입 화면이 서비스형 흐름에 맞는가
