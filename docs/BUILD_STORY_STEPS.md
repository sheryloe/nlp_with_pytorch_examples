## Step 1. 로컬 FastAPI + SQLite로 개인 가계부 MVP 만들기

### 메인 키워드

- 로컬 가계부 만들기
- FastAPI SQLite 가계부
- 개인 재무 대시보드

### 검색 의도

- 복잡한 배포 없이 바로 돌아가는 개인 가계부 MVP를 만드는 방법을 찾는 사람
- FastAPI와 SQLite 조합으로 빠르게 프로토타입을 만드는 흐름이 궁금한 사람

### 메타 제목

- 로컬 FastAPI + SQLite로 개인 가계부 MVP를 먼저 만든 이유

### 메타 설명

- 처음부터 배포와 인증까지 얹지 않고, FastAPI와 SQLite로 개인 가계부 MVP를 먼저 만든 이유와 실제 구현 포인트를 정리합니다.

### 도입부

사이드 프로젝트로 가계부를 만들 때 가장 자주 하는 실수가 있습니다. 처음부터 로그인, 배포, 외부 접속, 사용자 분리, 모바일 대응까지 한 번에 붙이려는 것입니다. 그렇게 시작하면 기능을 만들기도 전에 운영 문제에 먼저 지칩니다. 그래서 Donggri Ledger의 첫 단계는 아주 단순했습니다. “내가 실제로 매일 입력할 가계부 흐름이 맞는가”만 검증해보자는 것이었습니다. 이 단계에서는 기술 스택의 화려함보다 입력 속도, 목록 확인, 월별 요약, 데이터 저장 같은 기본 체감이 더 중요했습니다.

### 본문

첫 MVP는 FastAPI와 SQLite 조합으로 시작했습니다. 이유는 명확했습니다. FastAPI는 API와 화면을 빠르게 붙여보기 좋고, SQLite는 별도 서버 없이 바로 저장이 가능하기 때문입니다. 개인 가계부 MVP에서 필요한 건 수십 명 동시 접속이 아니라 “월급 1건, 식비 1건을 넣었을 때 내가 생각한 흐름과 똑같이 보이는가”입니다. 실제로 이 단계에서는 자산, 거래, 카테고리, 예산 정도만 잡고 월별 합계 카드와 거래 목록을 빠르게 확인하는 데 집중했습니다.  
  
중요했던 포인트는 완성도가 아니라 검증 순서였습니다. 먼저 자산을 하나 만들고, 수입과 지출을 넣고, 월 합계가 바로 갱신되는지 보았습니다. 프로그램을 껐다 켜도 데이터가 유지되는지, 거래 입력이 불필요하게 복잡하지 않은지, 실제 통장 잔액 감각과 큰 차이가 없는지를 체크했습니다. 이 과정 덕분에 “서비스형으로 가기 전에 무엇이 핵심 기능인지”를 먼저 알 수 있었습니다. 만약 이 단계 없이 바로 배포 구조로 갔다면, 실제로는 매일 쓰지도 않을 화면에 인증과 인프라만 얹고 있었을 가능성이 컸습니다.

### 코드 예시

```python
from fastapi import FastAPI

app = FastAPI()

@app.get("/api/transactions")
def get_transactions():
    return [
        {"date": "2026-03-01", "type": "income", "amount": 3200000},
        {"date": "2026-03-02", "type": "expense", "amount": 12800},
    ]
```

### 체크 포인트

- 수입 / 지출 입력 후 목록이 즉시 반영되는가
- 월별 합계가 사용자의 체감과 크게 다르지 않은가
- 데이터가 로컬 DB에 안정적으로 저장되는가
- 입력 폼이 너무 많은 선택지를 강요하지 않는가

### FAQ

- 왜 처음부터 Supabase로 시작하지 않았나요  
  처음엔 운영보다 사용 흐름 검증이 더 중요했기 때문입니다. 로컬 MVP가 있어야 진짜 필요한 기능이 보입니다.
- SQLite가 부족하지 않았나요  
  개인 1인용 MVP 단계에서는 충분했습니다. 부족해진 시점이 오히려 다음 구조 전환의 기준이 됐습니다.

### 마무리

이 Step의 핵심은 “빨리 만들었다”가 아니라 “무엇을 먼저 검증해야 하는지 순서를 맞췄다”에 있습니다. 가계부는 결국 매일 쓰는 도구라서, 첫 단계에서는 인프라보다 입력 감각이 훨씬 중요했습니다.

## Step 2. 카드 결제예정, 고정지출, 투자까지 붙여 실사용 가계부로 넓히기

### 메인 키워드

- 카드 결제예정 가계부
- 투자 포트폴리오 가계부
- 고정지출 예산 관리

### 검색 의도

- 카드 결제일과 실제 출금일이 다른 구조를 가계부에 어떻게 반영할지 궁금한 사람
- 소비, 예산, 투자 흐름을 한 앱에서 관리하는 방법을 찾는 사람

### 메타 제목

- 카드 결제예정과 투자 포트폴리오를 붙여 가계부를 실사용 단계로 확장한 방법

### 메타 설명

- 카드 결제일, 고정지출, 예산, 투자 포트폴리오를 한 흐름으로 묶어 개인 가계부를 실사용 단계로 확장한 설계 방법을 정리합니다.

### 도입부

가계부 MVP를 며칠만 써보면 바로 보이는 빈칸이 있습니다. 카드 결제는 쓴 날과 실제 돈이 빠지는 날이 다르고, 고정지출은 매달 반복되며, 투자 자산은 일반 소비와 분리해서 보기 어렵다는 점입니다. 이 세 가지가 빠지면 가계부는 단순 기록장에 머물고, 실제 생활비 관리 도구로는 부족해집니다. 그래서 두 번째 단계부터는 “입력 가능한 앱”이 아니라 “생활비를 실제로 통제할 수 있는 앱”으로 확장하는 작업이 필요했습니다.

### 본문

가장 먼저 손본 건 카드 결제 흐름이었습니다. 일반 가계부는 카드 사용일에 바로 돈이 빠지는 것처럼 보여주지만, 현실은 다릅니다. 그래서 카드 자산에는 결제일과 결제통장을 연결하고, 거래 입력 시에는 사용 내역과 정산 예정 내역을 분리해서 보도록 설계했습니다. 이 방식의 장점은 사용자는 간단히 카드 자산만 선택하면 되고, 실제 계산은 시스템이 일관되게 맡을 수 있다는 점입니다.  
  
고정지출과 예산 기능도 같은 맥락이었습니다. 월세, 통신비, 구독료처럼 매달 반복되는 비용을 미리 등록해두면 “이번 달 진짜 여유 자금”을 훨씬 빨리 파악할 수 있습니다. 여기에 카테고리별 예산까지 붙이면 단순 기록이 아니라 소비 속도를 조절하는 도구가 됩니다. 투자 포트폴리오를 같은 화면에 넣은 것도 중요한 변화였습니다. 소비는 소비 앱, 투자는 투자 앱으로 나누면 편할 것 같지만, 결국 내 자산은 한 덩어리입니다. 그래서 투자 계좌도 자산 모델 안에 두고, 원금/평가금/수익률을 가계부 흐름과 함께 보게 만들었습니다.

### 코드 예시

```json
{
  "name": "메인 카드",
  "type": "card",
  "card_settlement_day": 5,
  "card_settlement_asset_id": 2
}
```

### 체크 포인트

- 카드 사용 직후 현금 잔액이 즉시 빠지지 않는가
- 이번 달 카드결제예정 금액이 따로 보이는가
- 고정지출과 예산이 동시에 관리되는가
- 투자 원금, 평가금, 수익률이 일관되게 계산되는가

### FAQ

- 카드 결제예정을 거래와 따로 분리한 이유는 무엇인가요  
  사용일과 출금일이 다르기 때문입니다. 이 차이를 무시하면 월별 잔액 감각이 쉽게 무너집니다.
- 투자를 왜 같은 가계부 안에 넣었나요  
  생활비와 투자 자산을 분리해서 보면 전체 자산 흐름을 놓치기 쉽기 때문입니다.

### 마무리

이 단계부터 가계부는 단순 입력 도구를 넘어서 “이번 달에 내가 실제로 어떤 돈의 흐름 속에 있는가”를 보여주는 관리 도구가 되기 시작했습니다. 실사용 단계의 차이는 결국 현실 반영력에서 나옵니다.

## Step 3. GitHub Pages 랜딩과 SEO 구조로 프로젝트 설명력을 높이기

### 메인 키워드

- GitHub Pages SEO
- 프로젝트 소개 페이지 만들기
- 서비스 랜딩 페이지 예시

### 검색 의도

- GitHub 저장소를 단순 코드 모음이 아니라 소개 페이지로 보이게 만들고 싶은 사람
- 검색 유입을 받을 수 있는 프로젝트 랜딩 구조가 궁금한 사람

### 메타 제목

- GitHub Pages로 프로젝트 랜딩 페이지와 검색 유입 구조를 만든 과정

### 메타 설명

- 코드 저장소만으로는 설명하기 어려운 프로젝트를 GitHub Pages와 SEO 구조로 보완해, 소개 페이지와 검색 유입 구조를 함께 만든 과정을 정리합니다.

### 도입부

프로젝트가 어느 정도 커지면 “기능이 많아졌다”보다 “처음 보는 사람이 뭘 만든 건지 모르겠다”는 문제가 더 커집니다. README만으로는 한계가 있고, 저장소만 열어서는 제품의 느낌이 잘 전달되지 않습니다. 특히 사이드 프로젝트는 검색 유입이 약하기 때문에, GitHub Pages를 이용한 랜딩 페이지를 따로 구성하는 것만으로도 첫 인상과 설명력이 크게 달라집니다. Donggri Ledger도 바로 이 지점에서 소개 페이지를 따로 설계하기 시작했습니다.

### 본문

랜딩 페이지를 만들 때 가장 먼저 본 것은 디자인보다 정보 구조였습니다. 첫 화면에서 “이 서비스가 어떤 문제를 해결하는지”, “무슨 기능이 핵심인지”, “실제 화면이 어떻게 생겼는지”를 5초 안에 보여줘야 했습니다. 그래서 GitHub Pages는 단순히 긴 글을 늘어놓는 문서 페이지가 아니라, 제품 소개와 구조 설명이 동시에 가능한 형태로 바꿨습니다.  
  
SEO 측면에서는 기본 메타 태그만 넣는 수준에서 끝내지 않았습니다. Open Graph, description, canonical, FAQ, sitemap, 구조화 데이터까지 같이 정리해 두면 검색엔진이나 공유 미리보기에서 메시지가 훨씬 안정적으로 전달됩니다. 그리고 무엇보다 중요한 건 스크린샷입니다. 사용자가 실제 화면을 보지 못하면, 설명이 아무리 좋아도 추상적으로 느껴집니다. 그래서 대형 스크린샷, 제품 소개 문구, 구조 카드 중심으로 레이아웃을 바꾸고, 저장소 README와 말투도 최대한 맞췄습니다.

### 코드 예시

```html
<meta name="description" content="Supabase와 Vercel로 운영하는 개인 가계부 서비스">
<meta property="og:title" content="Donggri Ledger">
<meta property="og:type" content="website">
<link rel="canonical" href="https://sheryloe.github.io/donggri_gagyeobu/">
```

### 체크 포인트

- GitHub Pages 첫 화면에서 제품 정체성이 바로 보이는가
- 검색엔진용 description과 제목이 정리되어 있는가
- 스크린샷이 작게 흩어지지 않고 핵심 화면으로 보이는가
- README와 GitHub Pages의 메시지가 충돌하지 않는가

### FAQ

- GitHub Pages를 꼭 써야 하나요  
  꼭은 아니지만, 이미 GitHub를 쓰고 있다면 비용과 연결 측면에서 가장 가볍게 시작할 수 있습니다.
- 디자인보다 SEO가 더 중요했나요  
  둘 다 중요하지만, 첫 단계에서는 “무엇을 만드는 프로젝트인지 이해되게 하는 것”이 먼저였습니다.

### 마무리

이 Step의 핵심은 보기 좋은 페이지를 만든 것이 아니라, 프로젝트가 검색과 공유의 맥락 안에서도 설명될 수 있게 만든 것입니다. 코드가 전부가 아닌 시점부터는 설명 구조도 기능의 일부가 됩니다.

## Step 4. Supabase + Vercel로 서비스형 구조로 전환하기

### 메인 키워드

- Supabase 가계부
- Vercel 개인 서비스 배포
- SQLite에서 Supabase로 이전

### 검색 의도

- 로컬 웹앱을 외부 접속 가능한 서비스형 구조로 바꾸고 싶은 사람
- Supabase와 Vercel 조합이 실제로 어떻게 역할 분리되는지 알고 싶은 사람

### 메타 제목

- 로컬 FastAPI + SQLite 가계부를 Supabase + Vercel 서비스형 구조로 옮긴 이유

### 메타 설명

- 집 PC를 켜두는 로컬 구조 대신 Supabase와 Vercel을 활용해, 인증과 데이터 분리까지 가능한 서비스형 가계부 구조로 전환한 과정을 정리합니다.

### 도입부

로컬에서 잘 돌아가는 앱과 외부에서 안정적으로 쓸 수 있는 서비스는 전혀 다른 문제입니다. 집 PC를 켜둬야 하거나, 포트와 네트워크를 직접 관리해야 하거나, 웹훅과 외부 접근을 따로 뚫어야 하는 구조는 오래가기 어렵습니다. 특히 사용자별 데이터 분리까지 필요해지면 SQLite 기반 단일 로컬 구조는 분명한 한계가 생깁니다. Donggri Ledger도 이 시점에서 큰 구조 전환을 결정했습니다.

### 본문

역할을 먼저 분리했습니다. 프론트는 Vercel, 인증과 데이터는 Supabase, 코드와 문서는 GitHub로 나눴습니다. 이 구조의 장점은 책임이 명확하다는 점입니다. Vercel은 정적 프론트를 빠르게 배포하고, Supabase는 Auth, Postgres, RLS, Edge Functions를 한 번에 관리합니다. 사용자가 늘지 않더라도 “외부에서 접속 가능하고, 사용자별 데이터가 섞이지 않으며, 배포가 단순하다”는 점만으로 구조 전환의 가치가 충분했습니다.  
  
전환 과정에서는 단순히 DB만 옮긴 것이 아니라 데이터 모델도 다시 봤습니다. 각 테이블에 `user_id` 개념을 넣고, `RLS`로 본인 데이터만 접근 가능하게 바꿨습니다. 기존 로컬 SQLite에서 가능했던 흐름을 그대로 유지하되, 서비스형 구조에서 위험한 부분은 정책으로 막아야 했습니다. 여기에 회원가입, 비밀번호 복구, 투자 시세 갱신은 Edge Functions로 분리해 운영 로직도 조금 더 명확하게 정리했습니다.

### 코드 예시

```sql
create policy "transactions_all_own" on public.transactions
for all to authenticated
using (user_id = auth.uid())
with check (user_id = auth.uid());
```

### 체크 포인트

- 회원가입과 로그인 흐름이 외부 배포 환경에서 동작하는가
- 사용자별 데이터가 절대 섞이지 않는가
- 카드 / 예산 / 투자 흐름이 기존 UX를 유지하는가
- GitHub push 후 Vercel 재배포가 자연스럽게 이어지는가

### FAQ

- 왜 백엔드 서버를 따로 유지하지 않았나요  
  운영 단순화가 더 중요했기 때문입니다. 작은 서비스에서는 인프라를 줄이는 것이 큰 장점이 됩니다.
- Supabase를 쓸 때 가장 중요했던 점은 무엇이었나요  
  인증보다도 `RLS`를 통한 데이터 분리가 핵심이었습니다.

### 마무리

이 단계는 기술 스택 교체가 아니라 운영 방식 전환에 가깝습니다. 로컬 툴에서 서비스형 앱으로 바뀌는 시점에는 “어디서 실행되느냐”보다 “누가 어떤 데이터에 접근하느냐”가 훨씬 중요해집니다.

### 메타 설명

- SMTP와 도메인 세팅 부담을 줄이기 위해 보안질문 기반 복구 구조를 도입하고, Supabase Edge Function 오류와 투자 시세 새로고침 흐름을 안정화한 과정을 정리합니다.

### 도입부

서비스형 구조로 바꾸고 나면 이제부터는 “기능이 있다”보다 “막히지 않는다”가 더 중요해집니다. 특히 회원가입, 비밀번호 찾기, 실시간 가격 새로고침은 사용자가 바로 체감하는 기능이라 작은 문제도 신뢰도에 크게 영향을 줍니다. Donggri Ledger에서도 이 단계가 가장 디버깅 밀도가 높았습니다. SMTP, JWT, API key, 환경변수, 함수 응답이 서로 얽혀 있었기 때문입니다.

### 본문

처음에는 이메일 기반 비밀번호 복구를 생각했지만, 초반 운영에서는 도메인과 SMTP 설정 부담이 꽤 컸습니다. 그래서 보안질문 기반 복구 흐름으로 방향을 틀었습니다. 이 방식은 정석적인 대규모 서비스 해법은 아니지만, 소규모 운영에서는 훨씬 단순하고 빠르게 안정화할 수 있다는 장점이 있었습니다. 회원가입 시 질문 3개와 답변을 저장하고, 비밀번호 찾기에서 해당 질문에 답한 뒤 비밀번호를 재설정하도록 흐름을 만들었습니다.  
  
Edge Function 디버깅도 중요한 포인트였습니다. `Invalid API key`, `column does not exist`, `non-2xx status code` 같은 오류는 한 번에 보면 막막하지만, 하나씩 분리하면 결국 설정 문제와 코드 문제로 나뉩니다. 그래서 함수 쪽에서는 에러 원문이 최대한 드러나게 하고, 프론트에서는 토큰 헤더와 호출 응답을 더 명확하게 정리했습니다. 투자 탭의 실시간 가격 새로고침도 같은 맥락입니다. 가격이 안 불러와지면 기능이 없는 것보다 더 답답하기 때문에, 호출 흐름과 에러 메시지를 먼저 정리해 신뢰도를 높였습니다.

### FAQ

- 왜 이메일 대신 보안질문을 택했나요  
  초기 운영에서 SMTP와 도메인 세팅 부담을 줄이는 것이 우선이었기 때문입니다.
