# Donggri Ledger Build Story Draft

Notion 하위페이지로 옮기기 위한 Step 1~5 초안입니다.  
구성 원칙은 `문제 -> 왜 그렇게 구현했는지 -> 실제 구현 -> 기능 확인 -> 다음 단계` 흐름으로 맞췄습니다.

## Step 1. 로컬 FastAPI + SQLite로 개인 가계부 MVP 만들기

### 추천 제목

- 로컬 FastAPI + SQLite로 개인 가계부 MVP를 만든 이유

### 검색 유입 키워드

- local ledger app
- fastapi budget tracker
- sqlite finance dashboard
- 로컬 가계부 만들기
- 개인 재무 대시보드

### 글 구조

#### 문제

- 처음부터 서비스형으로 가면 인증, 배포, 데이터 분리, 도메인, 운영까지 한 번에 터집니다.
- 그래서 제일 먼저 확인해야 할 건 “내가 실제로 계속 쓸 입력 흐름이 맞는가”였습니다.

#### 왜 이렇게 구현했는가

- FastAPI는 빠르게 API와 화면을 같이 붙여보기 좋습니다.
- SQLite는 별도 인프라 없이 즉시 시작할 수 있습니다.
- MVP 단계에서는 운영성보다 입력 속도와 화면 감이 더 중요했습니다.

#### 구현 포인트

- 자산, 거래, 카테고리, 예산의 기본 모델 설계
- 월별 요약 카드
- 거래 입력과 목록 확인
- 로컬 DB 저장

#### 기능 확인

- 수입 / 지출 입력이 즉시 반영되는가
- 월별 합계가 맞는가
- 자산 잔액 흐름이 자연스러운가

#### 다음 단계로 넘어간 이유

- 실제 생활비 흐름에는 카드 결제예정, 고정지출, 투자 요약이 빠지면 금방 불편해집니다.

## Step 2. 카드 결제예정, 고정지출, 투자까지 붙여 실사용 가계부로 넓히기

### 추천 제목

- 카드 결제예정과 투자 포트폴리오를 붙여 가계부를 실사용 단계로 확장한 방법

### 검색 유입 키워드

- credit card settlement planner
- fixed expense tracker
- portfolio tracker in ledger
- 카드 결제예정 가계부
- 투자 포트폴리오 가계부

### 글 구조

#### 문제

- 일반 가계부는 “카드 쓴 날”과 “실제 돈 빠지는 날”이 다르다는 점을 자주 놓칩니다.
- 또 투자 포트폴리오를 별도 앱으로 보면 전체 자산 흐름이 끊깁니다.

#### 왜 이렇게 구현했는가

- 카드 결제일과 결제통장을 자산 속성으로 분리하면 거래 입력은 단순해지고 계산은 일관됩니다.
- 투자 계좌도 자산 모델 안에 두면 전체 재무 상태를 한 화면에서 볼 수 있습니다.

#### 구현 포인트

- 카드 자산별 결제일 / 결제통장 연결
- 카드 사용 시 실제 출금과 결제예정 분리
- 고정지출 등록
- 예산 설정
- 투자 계좌, 종목, 수익률 요약

#### 기능 확인

- 카드 사용 직후 현금 잔액이 즉시 빠지지 않는가
- 이번달 카드결제예정이 요약 카드에 보이는가
- 고정지출과 예산이 동시에 관리되는가
- 투자 원금, 평가금, 수익률이 계산되는가

#### 다음 단계로 넘어간 이유

- 기능이 많아질수록 프로젝트를 외부에서 설명하는 페이지와 검색 유입 구조가 필요해졌습니다.

## Step 3. GitHub Pages 랜딩과 SEO 구조로 프로젝트 설명력을 높이기

### 추천 제목

- GitHub Pages로 프로젝트 랜딩 페이지와 검색 유입 구조를 만든 과정

### 검색 유입 키워드

- github pages seo
- software landing page example
- project showcase page
- 프로젝트 소개 페이지 만들기
- SEO 랜딩 페이지

### 글 구조

#### 문제

- 코드만 있는 저장소는 검색 유입도 약하고, 프로젝트를 처음 보는 사람이 무엇을 만든 건지 이해하기 어렵습니다.

#### 왜 이렇게 구현했는가

- GitHub Pages는 이미 저장소와 연결돼 있어 문서와 소개 페이지를 빠르게 붙일 수 있습니다.
- 소개 페이지는 실제 서비스 화면을 보여주고 검색어를 설명하는 공간으로 쓰기 좋습니다.

#### 구현 포인트

- 소개용 랜딩 페이지
- Open Graph 메타 태그
- 구조화 데이터
- 스크린샷 기반 섹션
- FAQ와 sitemap

#### 기능 확인

- GitHub Pages에서 스크린샷이 잘 뜨는가
- 검색엔진용 description, keywords, FAQ가 들어갔는가
- 저장소 README와 메시지가 어긋나지 않는가

#### 다음 단계로 넘어간 이유

- 로컬 실행 구조는 설명용으론 좋았지만, 실제 외부 접속과 운영에는 한계가 분명했습니다.

## Step 4. Supabase + Vercel로 서비스형 구조로 전환하기

### 추천 제목

- 로컬 FastAPI + SQLite 가계부를 Supabase + Vercel 서비스형 구조로 옮긴 이유

### 검색 유입 키워드

- supabase budget app
- vercel finance app
- migrate sqlite to supabase
- 가계부 supabase
- vercel 개인 서비스 배포

### 글 구조

#### 문제

- 로컬 서버를 계속 켜두는 방식은 외부 접근, 웹훅, 배포 유지가 힘듭니다.
- 여러 기기에서 접속하거나 사용자 분리를 하려면 구조를 바꿔야 했습니다.

#### 왜 이렇게 구현했는가

- Vercel은 프론트 배포가 빠르고 단순합니다.
- Supabase는 Auth, Postgres, RLS, Edge Function을 한 곳에서 관리하기 좋습니다.
- GitHub는 코드와 문서 이력을 안정적으로 남길 수 있습니다.

#### 구현 포인트

- 정적 웹 앱 빌드 구조
- Supabase schema와 RLS
- 사용자별 데이터 분리
- Edge Function 기반 회원가입 / 복구 / 시세 갱신
- Vercel 환경변수와 빌드 설정

#### 기능 확인

- 회원가입과 로그인 흐름이 외부 배포 환경에서 도는가
- 사용자별 데이터가 섞이지 않는가
- 카드 / 예산 / 투자 데이터가 기존 흐름대로 유지되는가

#### 다음 단계로 넘어간 이유

- 서비스형으로 옮긴 뒤에는 인증 UX와 운영 에러 메시지를 더 안정적으로 다듬을 필요가 생겼습니다.

## Step 5. 아이디 인증, 보안질문 복구, 실시간 시세 안정화까지 마무리하기

### 추천 제목

- 보안질문 기반 회원가입과 실시간 시세 새로고침을 안정화한 마지막 정리

### 검색 유입 키워드

- security question auth
- supabase edge function auth
- yahoo finance edge function
- 보안질문 비밀번호 찾기
- supabase edge function 오류 해결

### 글 구조

#### 문제

- SMTP와 도메인 연결은 초기 운영 부담이 컸습니다.
- Edge Function은 JWT, API key, 환경변수, 에러 본문 때문에 디버깅이 까다로웠습니다.

#### 왜 이렇게 구현했는가

- 이메일 기반 복구 대신 보안질문 기반 복구를 택해 운영 복잡도를 줄였습니다.
- 에러는 숨기지 말고 원문이 보이게 해야 다음 수정 속도가 빨라집니다.
- 실시간 가격 새로고침은 투자 화면 신뢰도를 결정하는 기능이라 가장 마지막에 안정화했습니다.

#### 구현 포인트

- 아이디 + 비밀번호 + 보안질문 3개 회원가입
- 비밀번호 찾기용 질문 불러오기와 재설정
- JWT verification 설정 정리
- API key / 프로젝트 매칭 점검
- 투자 시세 새로고침 호출 보강
- 에러 본문 노출 개선

#### 기능 확인

- 회원가입 성공
- 로그인 성공
- 비밀번호 재설정 성공
- 투자 가격 새로고침 성공
- 삭제 버튼, 에러 메시지 같은 마감 UX 정리

#### 다음 단계

- 반복 수입
- 계좌 간 이체
- 월마감 스냅샷
- 알림 센터
- 운영 대시보드
